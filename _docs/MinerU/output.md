# Output formats

MinerU normalises every backend (`pipeline`, `vlm`, `hybrid`, plus
office) to the same canonical intermediate — **middle JSON** — and then
renders that into the user-facing files.

## Output bundle (per document)

For input `doc.pdf`:

```
out/doc/
├─ doc.md                       # main markdown
├─ doc_content_list.json        # ordered content list (paragraphs, tables, figures…)
├─ doc_middle.json              # canonical intermediate; full layout tree
├─ doc_model.json               # raw model output (debug / re-rendering)
├─ doc_layout.pdf               # original PDF with detected bboxes drawn
├─ doc_spans.pdf                # original PDF with detected text spans drawn
├─ doc_origin.pdf               # copy of the original file
└─ images/                      # extracted figures, charts, diagrams
   └─ <hash>.jpg
```

A ZIP of all of the above is what `mineru-api` returns from `/tasks/{id}/result`.

## Middle JSON (`*_middle.json`)

The schema is shared across all backends. Top level:

```json
{
  "pdf_info": [
    {
      "page_idx": 0,
      "page_size": [W, H],
      "preproc_blocks": [...],
      "para_blocks": [...]
    }
  ]
}
```

Each `para_block` has:
- `bbox`: `[x1, y1, x2, y2]`
- `type`: `text` | `title` | `table` | `image` | `equation` | `list` | …
- `lines` / `cells` / `latex` depending on type
- `level` for titles (h1..h5)

This is what RAG pipelines should consume directly — it preserves layout,
ordering, and structure that markdown rendering flattens out.

## Markdown (`*.md`)

Rendered by `pipeline_middle_json_mkcontent.py` /
`vlm_middle_json_mkcontent.py` / `office_middle_json_mkcontent.py` (one per
backend family) via a shared `union_make` interface. Headers/footers
removed by default, tables rendered as raw HTML (preserves
colspan/rowspan), math in `$...$` / `$$...$$`, figures as `![](images/...)`.

## Content list (`*_content_list.json`)

Flat list of content items in reading order — easier for RAG chunking than
the nested middle JSON.

```json
[
  {"type": "title", "text": "Introduction", "level": 1, "page_idx": 0},
  {"type": "text",  "text": "MinerU is...", "page_idx": 0},
  {"type": "table", "html": "<table>…</table>", "page_idx": 1},
  {"type": "image", "img_path": "images/abc.jpg", "img_caption": "Fig 1", "page_idx": 2}
]
```

## Visualizations (`*_layout.pdf`, `*_spans.pdf`)

Generated by `mineru/cli/visualization.py` in a separate process pool
(parallel with main parsing). Draws coloured boxes over the original PDF
for visual QA.

- `_layout.pdf`: block-level bboxes (titles, paragraphs, tables, figures…)
- `_spans.pdf`: span-level bboxes (each detected text run)

Disable via `--no-visualization` or remove the `VisualizationContext` from
your script if calling programmatically.

## Reading order

MinerU sorts blocks into human reading order using a combination of
XY-cut, vertical clustering, and (in 3.0+) the VLM's understanding of
multi-column layouts. Header/footer detection + removal happens before
sorting so they don't fragment the body flow.
